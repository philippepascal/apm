## Spec

### Problem

`apm start <id>` currently provisions a worktree, claims the ticket, and prints
the worktree path. The supervisor agent is then responsible for doing the actual
implementation work — either inline or by spawning subagents via the Agent tool.

This creates several problems:
- Subagents spawned via the Agent tool run in the supervisor's process context, not in the worktree
- Every Bash call from a subagent requires individual user approval (permission noise)
- Compound shell commands (`cd $wt && cargo test`) trigger hardcoded security warnings that bypass the allow list
- Multiple parallel tickets compete for the supervisor's attention

A new flag `--spawn` adds the ability to launch a `claude` worker subprocess
inside the worktree. `apm start <id>` (without `--spawn`) retains its current
behaviour exactly. `apm start --spawn <id>` provisions the worktree, launches
the worker in the background, and returns immediately. The supervisor gets
control back and can start additional tickets or respond to the user. The worker
is responsible for committing, pushing, and opening a PR when done.

### Acceptance criteria

- [x] `apm start <id>` (no `--spawn`) behaves exactly as today: provision
  worktree, claim ticket, print worktree path, exit — no subprocess launched
- [x] `apm start --spawn <id>` provisions the worktree, launches a `claude`
  worker subprocess in the background, and returns immediately (non-blocking);
  prints the worker PID and log path before returning
- [x] By default `--spawn` does NOT pass `--dangerously-skip-permissions` to
  the worker; the worker runs with the project's `.claude/settings.json` allow
  list (which must include all commands the worker needs)
- [x] `--spawn --skip-permissions` (or `-P`) additionally passes
  `--dangerously-skip-permissions` to the worker for users who prefer full
  autonomy over the allow-list safety net
- [x] The worker is launched with `--print` (non-interactive) and receives
  `apm.worker.md` as system prompt
- [x] The ticket content (`apm show <id>`) is passed as the initial user
  message to the worker
- [x] Worker stdout+stderr are captured to `<worktree>/.apm-worker.log`
- [x] The worker is responsible for: committing changes, pushing the branch,
  opening a PR targeting `main`, and transitioning the ticket to `implemented`
  or `blocked`
- [x] `apm init` creates `apm.worker.md` alongside `apm.agents.md`
- [x] `apm.worker.md` is focused on worker-only concerns: implementing a
  ticket, committing, pushing, creating a PR, transitioning state, and the
  blocked protocol
- [x] `apm.agents.md` (supervisor) is updated to describe the new workflow:
  the entry point is a Claude Code supervisor session; `apm start --spawn <id>`
  is the primitive to hand a ticket to a background worker
- [x] The `APM_AGENT_NAME` for the worker is generated by `apm start` and
  passed as an environment variable to the subprocess

### Out of scope

- A standalone `apm work` command that orchestrates all ready tickets without a
  supervisor Claude session — that is a separate ticket once the worker model is proven
- Monitoring or tailing worker logs via an `apm` command
- Killing or restarting running workers
- Worker health checks or timeout enforcement
- SDK-based worker orchestration (claude CLI is the implementation target)
- Multi-turn worker conversations (one prompt → worker runs to completion)

### Amendment requests

- [x] Clarified blocking vs non-blocking: `apm start` is non-blocking — it
  launches the worker in the background and returns immediately so the supervisor
  can start more tickets or respond to the user. The worker owns push and PR
  creation. The previous spec contradicted itself (Problem said "returns
  immediately", acceptance criteria said "blocks"); resolved in favor of
  non-blocking.
- [x] `apm start` (without flags) retains current behaviour exactly; `--spawn`
  is the additive flag, not `--no-spawn` as a toggle-off
- [x] `--dangerously-skip-permissions` is opt-in via `--skip-permissions` (`-P`),
  not the default; workers run under the project allow list by default so users
  who have configured `.claude/settings.json` correctly do not need to weaken
  their safety posture
- [x] Clarified the intended entry point: `apm start --spawn` is a primitive
  for a Claude Code supervisor session, not a standalone orchestrator; the user
  opens a Claude Code session and directs the supervisor to start workers — a
  future `apm work` command (headless orchestration) is explicitly out of scope
  for this ticket

### Approach

**`apm/src/cmd/start.rs`**: add `--spawn` and `--skip-permissions` / `-P` flags.
After provisioning the worktree and claiming the ticket, if `--spawn` is set:

1. Generate `APM_AGENT_NAME` as `claude-MMDD-HHMM-XXXX`
2. Read `apm.worker.md` from the repo root
3. Get ticket content from the ticket struct already loaded
4. Build the `claude` command:
   - Always: `--print`, `--system <worker_md>`
   - If `--skip-permissions`: also `--dangerously-skip-permissions`
   - Positional: ticket content as the user message
5. Set `APM_AGENT_NAME` in the child environment
6. Redirect worker stdout+stderr to `<worktree>/.apm-worker.log`
7. Spawn detached (not waited on); print PID and log path; return

If `--spawn` is not set, exit after printing the worktree path as today.

**`apm/src/cmd/init.rs`**: create `apm.worker.md` during `apm init` using
`include_str!` on a new template file in the crate.

**`apm.worker.md` template** (worker-only instructions):
- Identity: read `APM_AGENT_NAME` from the environment
- Startup: `apm show <id>` (ticket number is in the user message)
- Implementation: code changes, tests, commit to branch
- Push: `git push origin <branch>`
- PR: `gh pr create --base main ...`
- Transition: `apm state <id> implemented`
- Blocked protocol: write questions to `### Open questions`, commit, push,
  `apm state <id> blocked`

**`apm.agents.md` update** (supervisor-only instructions):
- Clarify that the entry point is a Claude Code supervisor session
- For implementation: `apm start --spawn <id>` hands the ticket to a background
  worker and returns immediately; use `--skip-permissions` (`-P`) if the project
  allow list does not cover all worker commands
- Supervisor concerns: spec review, unblocking, PR review — not push or PR creation
