## Spec

### Problem

Two CLI help strings become stale after the T1 (4bee5771) and T3 (d8e2fa0e) redesign.

`apm instructions` (line 894 of `apm/src/main.rs` and the `PREAMBLE` constant in `apm/src/cmd/instructions.rs`) currently describes the command as emitting "a compact plain-text guide" that lists commands. After T1, the command calls `apm_core::instructions::generate()` and emits full APM system knowledge across five named sections: state machine, ticket format, shell discipline, session identity, and command reference. The about string and any surviving intro text must reflect this.

`apm prompt` (the `#[command(long_about = "...")]` block in `apm/src/main.rs`, lines 842–879) documents a flat 5-level cascade (levels 0–4) and shows a `--explain` sample with `prefix:` / `system prompt:` labels. After T3, the prompt composes three layers — layer 1 (dynamic apm instructions), layer 2 (project context file), layer 3 (role-file cascade) — and `format_provenance` outputs `layer 1:` / `layer 2:` / `layer 3:` labels. The long_about and its embedded `--explain` example must describe the three-layer model and match the T3 output format.

### Acceptance criteria

- [ ] `apm instructions --help` describes the output as APM system knowledge (state machine, ticket format, shell discipline, session identity, command reference) — not as a compact command list
- [ ] `apm prompt --help` does not describe the 0–4 cascade levels as top-level composition steps
- [ ] `apm prompt --help` describes three named layers: layer 1 = apm instructions (dynamic), layer 2 = project context file, layer 3 = role-file cascade
- [ ] The `--explain` sample in `apm prompt --help` shows `layer 1:`, `layer 2:`, `layer 3:`, and `skipped:` labels, matching the T3 `format_provenance` output format
- [ ] No help text retains the old `prefix:` or `system prompt:` explain labels
- [ ] The last example line in `apm prompt --help` reads `# show layer provenance` (not `# show cascade provenance`)
- [ ] Dead-code constants and helpers in `apm/src/cmd/instructions.rs` left orphaned by T1 (`PREAMBLE`, `render`, `render_compact_commands` and their unit tests) are removed if still present after T1 lands

### Out of scope

- Updating `format_provenance` output format or `PromptProvenance` struct — covered by T3 (d8e2fa0e)
- Changing `LEVEL_LABELS` in `apm-core/src/start.rs` (`per-agent file`, `transition.instructions`, etc.) — these remain accurate as sub-level labels within layer 3
- Generating the content of the five sections emitted by `apm instructions` — covered by T1 (4bee5771)
- Adding `--role` flag to `apm instructions` — covered by T1
- Updating test assertions in `apm-core/src/prompt.rs` (`explain_prefix_shown`) to check `layer 2:` — covered by T3 step 9
- Any behaviour change; this ticket is text-only

### Approach

#### 1. `apm/src/main.rs` — Instructions about string

#### 2. `apm/src/cmd/instructions.rs` — dead-code removal

If `PREAMBLE`, `render()`, and `render_compact_commands()` are still present (T1's spec leaves their removal ambiguous), remove them along with their unit tests (`render_includes_preamble`, `render_includes_command_name`, `render_no_ansi`, `render_compact_has_apm_prefix`, `render_compact_shows_about`, `render_compact_no_flags`, `render_compact_excludes_hidden`, `run_returns_ok`). #### 3. `apm/src/main.rs` — Prompt long_about

Replace the `#[command(long_about = "...")]` attribute on the `Prompt` variant (lines 842–879) with:

```
Print the system prompt that would be used if the ticket's current
command:start transition fired, then exit 0. The ticket is not modified
and no worker is spawned.

The prompt is composed from three layers, concatenated in order:

  Layer 1 — APM system knowledge
    Generated by `apm instructions --role <role>` (dynamic, role-scoped).
    Covers: state machine, ticket format, shell discipline, session identity,
    and command reference.

  Layer 2 — Project context
    The file at [agents].project in config.toml (typically apm.project.md).
    Omitted if the key is not set or the path is empty.

  Layer 3 — Role instructions (cascade, highest priority wins)
    0. .apm/agents/<agent>/apm.<role>.md     (per-agent file)
    1. transition.instructions               (from workflow.toml)
    2. profile.instructions                  (from [worker_profiles.*])
    3. [workers].instructions                (global fallback)
    4. built-in default for known agents
    5. error — no instructions found

Without a ticket ID, --agent and --role determine the mode:
  • Both provided: print the system prompt for that agent+role (layer 3 levels
    1 and 2 are skipped — no ticket means no transition or profile context).
  • Either missing: print the agents and roles discovered in .apm/agents/.

--agent and --role override the resolved values for inspection only;
they do not affect the ticket or any config.

--explain prints a compact provenance table instead of the prompt text:

  layer 1:         apm instructions (dynamic, role: worker)
  layer 2:         .apm/agents/default/apm.project.md
  layer 3:         .apm/agents/claude/apm.worker.md  (level 0 — per-agent file)
  skipped:         level 1 (transition.instructions — none set)
                   level 2 (profile.instructions — none set)
                   level 3 (workers.instructions — none set)

Exits non-zero with a clear message when no instructions can be resolved.

Examples:
  apm prompt                                       # list available agents and roles
  apm prompt --agent default --role worker         # base prompt for default/worker
  apm prompt --agent default --role worker --explain  # provenance without a ticket
  apm prompt ba121f45                              # resolved agent + role for ticket
  apm prompt ba121f45 --agent pi                   # inspect pi-agent prompt
  apm prompt ba121f45 --role spec-writer           # inspect spec-writer role
  apm prompt ba121f45 --explain                    # show layer provenance
```

The `--explain` sample is derived from T3 step 7's format_provenance spec; if T3's actual output format differs in spacing or field order, match the actual runtime output.
