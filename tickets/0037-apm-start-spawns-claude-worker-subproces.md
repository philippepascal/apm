## Spec

### Problem

`apm start <id>` currently provisions a worktree, claims the ticket, and prints
the worktree path. The supervisor agent is then responsible for doing the actual
implementation work — either inline or by spawning subagents via the Agent tool.

This creates several problems:
- Subagents spawned via the Agent tool run in the supervisor's process context, not in the worktree
- Every Bash call from a subagent requires individual user approval (permission noise)
- The supervisor blocks while waiting for subagents to finish
- Compound shell commands (`cd $wt && cargo test`) trigger hardcoded security warnings that bypass the allow list
- Multiple parallel tickets compete for the supervisor's attention

The right model: `apm start <id>` provisions the worktree AND launches a `claude`
subprocess inside it, then returns immediately. The supervisor gets control back
and can start additional tickets. Each worker runs fully autonomously in its own
directory with its own permission context.

### Acceptance criteria

- [ ] `apm start <id>` provisions the worktree, then launches a `claude` subprocess inside it and **blocks** until the subprocess exits
- [ ] The worker is launched with `--dangerously-skip-permissions` so it runs autonomously without prompting
- [ ] The worker is launched with `--print` (non-interactive) and receives `apm.worker.md` as system prompt
- [ ] The ticket content (`apm show <id>`) is passed as the initial user message
- [ ] Worker stdout+stderr are streamed to the terminal AND captured to `<worktree>/.apm-worker.log`
- [ ] After the worker exits, `apm start` inspects the ticket state:
  - If state is `implemented`: push the branch and create a PR targeting `main`
  - If state is `blocked`: push the branch (so the supervisor can read the questions); do not create a PR
  - Any other exit state: print a warning and exit without pushing
- [ ] `apm start --no-spawn <id>` provides the previous behavior (provision worktree, print path, no subprocess)
- [ ] `apm init` creates `apm.worker.md` alongside `apm.agents.md`
- [ ] `apm.worker.md` is focused on worker-only concerns: implementing a ticket, committing changes, state transitions, the blocked state protocol — **no push, no PR creation**
- [ ] `apm.agents.md` (supervisor) is updated to describe the new workflow: run `apm start <id>` per ticket; it blocks and handles push+PR automatically
- [ ] The `APM_AGENT_NAME` for the worker is generated by `apm start` and passed as an environment variable to the subprocess

### Out of scope

- Monitoring or tailing worker logs via an `apm` command
- Killing or restarting running workers
- Worker health checks or timeout enforcement
- SDK-based worker orchestration (claude CLI is the implementation target)
- Multi-turn worker conversations (one prompt → worker runs to completion)
- The worker pushing branches or creating PRs (those are `apm start`'s responsibility)

### Approach

**`apm/src/cmd/start.rs`**: After provisioning the worktree and claiming the ticket,
check if `--no-spawn` is set. If not:

1. Generate `APM_AGENT_NAME` as `claude-MMDD-HHMM-XXXX`
2. Read `apm.worker.md` from the repo root
3. Get ticket content from the ticket struct already loaded
4. Build the claude command with `--dangerously-skip-permissions`, `--print`,
   `--system <worker_md>`, and the ticket content as the user message
5. Spawn the subprocess with stdout+stderr piped; stream both to the terminal and
   to `<worktree>/.apm-worker.log`; **wait** for the process to exit
6. After exit, re-read the ticket from the branch to determine final state:
   - `implemented` → `git push origin <branch>` then `gh pr create --base main`
   - `blocked` → `git push origin <branch>` (no PR)
   - anything else → print warning, exit non-zero

**`apm/src/cmd/init.rs`**: Create `apm.worker.md` during `apm init` using `include_str!`
on a new `apm.worker.md` template file in the repo.

**`apm.worker.md` template** (worker-specific instructions):
- Identity: set `APM_AGENT_NAME` from the environment variable provided by `apm start`
- Startup: `apm show <id>` to read the full spec (the ticket number is in the user message)
- Implementation: code changes, `cargo test --workspace`, commit changes locally
- **No push, no PR** — `apm start` handles those after the worker exits
- Blocked protocol: write questions to `### Open questions`, run `apm state <id> blocked`, exit
- Completion: run `apm state <id> implemented`, exit — push and PR are handled for you

**`apm.agents.md` update** (supervisor-specific):
- Streamline to supervisor concerns only: spec review, unblocking, PR review
- Replace implementation workflow with: "Run `apm start <id>` — this launches a worker, waits for it, then pushes and creates the PR automatically"
- The supervisor no longer needs to push branches or create PRs manually
