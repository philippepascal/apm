## Spec

### Problem

During implementation an agent may hit a decision point that requires supervisor
input — a design ambiguity, a scope question, or an external dependency. Without
a formal state for this, the agent is forced to either guess, abandon the ticket
(rolling back to `ready` and losing context), or leave the ticket stuck in
`in_progress` indefinitely with no signal to the supervisor.

### Acceptance criteria

- [x] `blocked` state exists in the default `apm.toml` generated by `apm init` with `actionable = ["supervisor"]`
- [x] Transition `in_progress → blocked` is defined (actor: agent) with a label indicating the agent wrote questions in `### Open questions`
- [x] Transition `blocked → ready` is defined (actor: supervisor)
- [x] `apm start` is restricted to states with a `command:start` trigger (i.e. `ready`), not all agent-actionable states — prevents accidentally transitioning spec-writing states to `in_progress`
- [x] `apm.agents.md` documents `blocked` state behaviour: agent writes questions then blocks; on unblock, agent resumes via `apm start` reusing the existing worktree
- [x] `STATE-MACHINE.md` reference implementation includes `blocked` in the states table

### Out of scope

- Auto-notifying the supervisor on block
- Timeout or escalation logic
- Blocking from states other than `in_progress`

### Approach

Add `blocked` state between `in_progress` and `ready` in `apm.toml` and
`default_config()`. Add `in_progress → blocked` transition (actor: agent).
Add `blocked → ready` transition (actor: supervisor). Restrict `apm start`
to check for `command:start` trigger on the current state rather than using
the general `actionable_states_for("agent")` list. Update `apm.agents.md`
and `STATE-MACHINE.md`.
